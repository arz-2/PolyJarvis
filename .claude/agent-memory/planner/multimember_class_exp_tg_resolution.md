For polymer classes whose `experimental_tg_K` in polymer_rules.json is a **dict of multiple members** (e.g. PVNL = {PVA:303, PVC:354, PVAc:304}), `make_deterministic_plan.py` cannot resolve which member the SMILES is and picks the wrong scalar for the `tg` stage `t_range_brackets_exp_tg` criterion (it grabbed 304/PVA for a PVC run, *CC(Cl)*).

**Why:** The script keys off `polymer_class`, not the specific monomer; the class dict has no SMILES→Tg mapping. The Tg sweep *range* (class-wide, e.g. 150-550 K) usually still brackets the true member Tg, so the run wouldn't fail — but the recorded criterion and any D-06 gate would reference the wrong number.

**How to apply:** On any reasoned plan for a multi-member class (PVNL, PDIE, and check others like POXI/PEST that list multiple examples), identify the specific member from the SMILES, then (1) set `planned_stages[tg].success_criteria.t_range_brackets_exp_tg` to that member's Tg, and (2) pin `decided_params.experimental_tg_K` (scalar) + `exp_density_gcm3` so downstream consumers use the member value, not the generic class dict. Record the member identification in `assumptions` as the key judgement call. Known members: PVC=354 K / 1.38 g/cm3, PVA=303 K / 1.26, PVAc=304 K / 1.19.

**Even in a pure deterministic (confidence=high) PDIE plan, the script mis-pins the member.** Confirmed 2026-06-18 on the corrected cis-PBD1 (true C4 `*C/C=C\C*`): `make_deterministic_plan.py` left `experimental_tg_K`/`exp_density_gcm3` null AND set the tg stage `t_range_brackets_exp_tg` to 200 (PI), not 181 (PBD). Fix is a member-disambiguation edit on the deterministic plan (does NOT flip plan_mode — pinning a member value the script structurally can't resolve is not re-reasoning a decision): set `decided_params.experimental_tg_K`=181, `exp_density_gcm3`=0.90, the tg bracket to 181, and record the C4-vs-C5 identification in `assumptions`. Plan stays deterministic/high/approved.

**PKTN is another multi-member class (PEEK/PEK/PEKK), confirmed 2026-06-18 on PEEK1.** make_deterministic_plan.py left `experimental_tg_K`/`exp_density_gcm3`/`exp_K_GPa` null and set the tg bracket to 433 (PEKK), not 418 (PEEK). Disambiguate by counting ether O and ketone C=O per repeat: PEEK = E-E-K (2 ether O + 1 ketone, `*Oc1ccc(Oc2ccc(C(=O)c3ccc(cc3)*)cc2)cc1`) = 418 K / 1.32 g/cm3 / K 4.0-7.0 GPa; PEK = E-K (1 ether) = 440 K; PEKK = E-K-K (2 ketones) = 433 K. Fix is the member-disambiguation edit (stays deterministic/high/approved): pin scalar `experimental_tg_K`/`exp_density_gcm3`/`exp_K_GPa` and set the tg bracket to the member value.

**Resolution must ALSO handle "SMILES matches NO listed member" — not just "pick the right one of N."** Confirm member identity by counting backbone atoms, do not key off run_name. PDIE example (cis-PBD1, 2026-06-18): run_name said "PBD" but SMILES `*CC/C=C\CC*` is a C6 diene with a 4-methylene spacer per C=C (-CH2-CH2-CH=CH-CH2-CH2-). Tabulated PDIE members: PBD = C4 `*C/C=C\C*` (181 K), cis-PI = C5 with pendant methyl `*CC(C)=CC*` (200 K). The C6 SMILES is neither, so borrowing 181/200 would validate the sim against the wrong polymer. Correct move: reasoned override (flip plan_mode, keep class confidence=high — see [[reasoned_override_confidence]]); run `estimate_tg_group_contribution.py` (use `/home/alexzhao/miniforge3/envs/mol-builder/bin/python` — base python lacks rdkit) on the ACTUAL SMILES. If estimate is `very_low` (>30% unmatched — happens because C=C carbons don't match the CH2 motifs), leave global T-defaults unchanged, set `experimental_tg_K`/`exp_density_gcm3` = null (honest), set dominant uncertainty `temperature_parameters_unvalidated` + `reduction_probe: literature_anchor`, and use the estimate (marked very_low) only as the `t_range_brackets_exp_tg` reference — never a tabulated member's value.
