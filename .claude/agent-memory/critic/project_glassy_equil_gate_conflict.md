Glassy high-MW polymer plans (regime=glassy & DP>=30) hit a conflict at the equilibration gate that is now RESOLVED in policy.

**History (round-1 escalate, PVC1 2026-06-20):** `policies.equilibration.require` was once an absolute "check_equilibration_comprehensive overall_pass=True before ANY property extraction" with NO glassy carve-out. For glassy DP>=30, overall_pass embeds a melt-NVT C(t) full-decay sub-check whose tau_relax (~3.6e9 ps for DP=60) is physically unattainable. A planner revision demoting overall_pass to advisory was scientifically sound but overrode a FIXED require, so the Critic escalated (could neither approve an override of fixed policy nor revise an identical plan back).

**Resolution (out-of-band, user-authorized 2026-06-20):** `decision_policy.json:policies.equilibration` now has a `require_glassy` field. The `require` clause reads "...EXCEPT glassy DP>=30 systems, which use require_glassy." `require_glassy` gates on: density plateau in experimental range AND density homogeneity CV<0.25 AND P2 nematic<0.10 AND energy drift/SEM within bounds; the four melt chain-self-diffusion metrics (C(t) decay, MSD diffusivity, MSID slope, Rg chain-chain CV) are ADVISORY, and overall_pass may be False on them without blocking property extraction. `rationale_glassy` documents the tau_relax infeasibility.

**How to apply (current rule):** When a reasoned plan for regime=glassy & DP>=30 demotes overall_pass to advisory AND its equil/equil-check success_criteria match require_glassy verbatim (four structural gates + four advisory diffusion metrics, rubbery path unchanged), the verdict is APPROVE — the framework now sanctions the demotion; it is no longer a Critic waiver. Verify regime via experimental_tg_K>300 (or measured Tg>300) and DP>=30. If a glassy plan demotes a gate that is NOT covered by require_glassy (e.g. demotes density homogeneity, or P2), that is still a finding. Determining glassy: PE1, PMMA1, PLA1 are the other glassy revision polymers expected to hit this path — approve them on the same basis. See [[born-method-implementation]] for the glassy K_T path that consumes the accepted npt_prod300 data.

**Rubbery trap — require_glassy does NOT cover rubbery melts (PE2/PHYC, 2026-06-22).** require_glassy is gated on `regime=glassy AND DP>=30`. A rubbery system (is_glassy=False, e.g. PE: exp Tg~195K, T_workflow=300K>Tg) that demotes overall_pass to advisory — gating only on density/homogeneity/P2 and treating C(t)/MSD/Rg as advisory — is applying the glassy carve-out logic the framework never authorized for rubbery. The `require` clause is UNCONDITIONAL for non-glassy ("overall_pass=True before ANY property extraction"); the ONLY exception is require_glassy. No `require_rubbery` clause exists. This is a HARD require violation. **In-pipeline fix? No** — the plan documents tau_relax~88ns (DP-120 PE melt); overall_pass is physically unreachable and extend-1-2ns can't close an 88ns gap, so keeping it gating = run can never pass, while demoting it = framework change (needs a require_rubbery amendment, user-authorized, out of planner/critic scope). Hard require violation + no in-pipeline fix = **ESCALATE in round 1**. Note: PE1 (replicate 1) was apparently approved on this same structure earlier in the campaign — if so that approval was non-conforming; PE2 onward enforce the rubbery gap. Same escalate/no-fix structure as the original PVC1 glassy conflict, just for the still-unsanctioned rubbery branch. A user-authorized require_rubbery clause (mirroring require_glassy) would resolve the whole PE/PP/PIB replicate campaign at once.

**Inverse trap — DP<30 cannot borrow require_glassy via a class field (PEEK2/PKTN, 2026-06-22).** require_glassy is gated STRICTLY on DP>=30. A reasoned plan at DP<30 (PEEK2 was DP=15) that demotes C(t)/MSD to advisory by citing a polymer_rules.json CLASS field (e.g. `ct_gate_reliable=false`, `_ct_note` "aromatic backbone — C(t) undefined") is a HARD require violation, NOT an approve. A class field is data, not an amendment to the fixed decision_policy.json; only require_glassy (DP>=30) sanctions the demotion. The argument "the metric is undefined for this backbone" may be scientifically true but is still not a sanctioned exception in the framework — the Critic enforces the framework, not the science directly. This is REVISE (not escalate) when the fix is in-pipeline: raise DP>=30, which simultaneously clears system_size.require (Fox-Flory DP>=20 for tg) AND legalizes the C(t)/MSD demotion under require_glassy. Escalate only if the planner returns round 2 still at DP<30 pleading budget (a budget/DP tradeoff needs USER approval, not a self-declared class exception). Distinguish from the legit case above purely by DP: DP>=30 demotion = approve; DP<30 demotion = finding.
